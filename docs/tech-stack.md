# Tech Stack — Book2Screen (Frontend)

Документ фіксує технологічний стек клієнтської частини проєкту **Book2Screen** та обґрунтовує вибір кожного інструмента з огляду на вимоги проєкту.

**Про проєкт:** Book2Screen — система відгуків і рейтингів книг та їх екранізацій. Користувач може порівнювати книгу з її фільмом/серіалом на основі рейтингів і відгуків інших клієнтів.

---

## 1. Нефункціональні вимоги (NFR), що вплинули на вибір

Перед вибором стеку команда зафіксувала ключові нефункціональні вимоги:

| Вимога | Опис |
|--------|------|
| **Швидкодія UI** | Миттєва реакція на дії користувача (фільтрація, пошук, перехід між сторінками) без перезавантаження |
| **Time-to-Market** | Обмежений термін MVP — потрібна швидкість розробки та низький поріг входу |
| **Підтримуваність** | Малий склад команди → код має бути читабельним, типізованим, з чіткою структурою |
| **Scalability** | Легкість додавання нових сторінок та компонентів без переписування архітектури |
| **Кросбраузерність / Адаптивність** | Підтримка сучасних браузерів та різних роздільних здатностей (desktop / tablet) |
| **Stateless-авторизація** | Використання JWT — сервер не зберігає сесії у пам'яті, що спрощує масштабування backend |

## 2. Технологічний стек

### 2.1. Мова та збірник

#### TypeScript `~5.9.3`
**Обґрунтування:** Статична типізація дозволяє ловити помилки ще на етапі розробки (а не в рантаймі у браузері користувача), що критично важливо для малої команди — ментор/тіммейти одразу бачать, які поля та типи очікуються в API-контрактах. TypeScript також значно покращує автодоповнення в IDE, що прискорює розробку. Використовується в strict-режимі (`noUnusedLocals`, `noUnusedParameters`, `strict: true`).

#### Vite `^8.0.3`
**Обґрунтування:** Сучасний build-tool з нативною підтримкою ES-модулів. Hot Module Replacement (HMR) працює майже миттєво, що критично для швидкості розробки. На відміну від Webpack, не потребує складної конфігурації «з коробки». Підтримує TypeScript та Vue без додаткових плагінів.

---

### 2.2. Framework та клієнтська частина

#### Vue 3 `^3.5.31` + Composition API (`<script setup>`)
**Обґрунтування:** Vue 3 обрано як основний UI-фреймворк через:
- **Низький поріг входу** — простіший синтаксис порівняно з Angular, менше boilerplate-коду ніж у React.
- **Composition API** — дозволяє виносити логіку в перевикористовувані composables (`use*`-функції), що підвищує підтримуваність коду.
- **Реактивність «з коробки»** — `ref`, `computed`, `watch` дають миттєве оновлення UI при зміні даних, що задовольняє вимогу швидкодії.
- **SFC (Single File Component)** — компонент, шаблон і стилі в одному файлі, що зручно для Code Review.

#### Vue Router `^4.6.4`
**Обґрунтування:** Офіційна бібліотека маршрутизації для Vue. Забезпечує роботу SPA без перезавантаження сторінки (вимога швидкодії). Підтримує параметризовані маршрути (`/item/:id`), що використовується на сторінці деталей порівняння.

#### Pinia `^3.0.4` (State Management)
**Обґрунтування:** Офіційно рекомендований Vue-командою state manager (наступник Vuex). Обрано замість альтернатив з наступних причин:
- **TypeScript-підтримка «з коробки»** — повне автодоповнення типів store-ів без додаткових типізацій.
- **Composition API-стиль** — store виглядає як звичайний composable (`defineStore` з функцією setup), що консистентно з рештою коду.
- **Менше boilerplate** ніж Vuex — немає mutations, actions та getters як окремих понять.
- **Для Book2Screen MVP** використовується для керування станом авторизації користувача (`useUserStore` в `src/state/user.ts`) — зберігання JWT-токена, email, статусу `isAuthenticated`.

---

### 2.3. HTTP-клієнт

#### Axios `^1.15.0`
**Обґрунтування:** Обрано замість нативного `fetch` через:
- **Interceptors** — дозволяють централізовано додавати JWT-токен до кожного запиту (`Authorization: Bearer <token>`) та централізовано обробляти помилки (автоматичний logout при 401).
- **Автоматичне перетворення JSON** — не потрібно вручну викликати `.json()`.
- **Таймаути та скасування запитів** — простіше керувати життєвим циклом запитів.
- **Уніфікована обробка помилок** — всі HTTP-помилки проходять через один `response interceptor` (див. `src/services/api.ts`).

---

### 2.4. Стилізація

#### Vanilla CSS (scoped styles у SFC) + CSS Custom Properties
**Обґрунтування:** Для MVP обрано вбудовані можливості Vue SFC (`<style scoped>`), оскільки:
- **Не потребує додаткових бібліотек** — менший розмір bundle, швидше завантаження.
- **Scoped-стилі** автоматично ізолюють CSS кожного компонента — немає конфліктів класів.
- **CSS-змінні** (`--pink-light`, `--accent`, `--dark-card`) у `App.vue` задають єдину палітру проєкту та легко міняються централізовано.
- Для MVP з обмеженою кількістю сторінок UI-бібліотека (Material UI / Vuetify) створила б зайвий overhead.

---

### 2.5. Інструменти якості коду

#### ESLint `^10.1.0` + конфіги `@vue/eslint-config-typescript` та `@vue/eslint-config-prettier`
**Обґрунтування:** ESLint — галузевий стандарт лінтингу для JS/TS. Офіційні Vue-конфіги забезпечують правила, специфічні для Vue 3 (`plugin:vue/vue3-essential`). Додатково налаштовано правило `id-denylist` проти «сміттєвих» назв (`data`, `info`, `temp`).

#### Prettier `^3.8.1`
**Обґрунтування:** Автоматичний форматтер коду. Усуває суперечки в команді про стиль (де ставити пробіли, лапки, крапки з комою). Інтегрований з ESLint через `@vue/eslint-config-prettier/skip-formatting`, щоб правила не конфліктували. Налаштований через `.prettierrc`: `singleQuote: true`, `semi: true`, `tabWidth: 2`, `printWidth: 120`.

#### vue-tsc `^3.2.5`
**Обґрунтування:** TypeScript-компілятор, який розуміє `.vue`-файли. Використовується в скрипті `npm run build` для валідації типів у production-збірці.

## 4. Інтеграції та зовнішні залежності

### 4.1. Backend API
- **Протокол:** REST (HTTP + JSON)
- **Авторизація:** JWT-токен у заголовку `Authorization: Bearer <token>`
- **Базова URL-адреса:** задається через змінну оточення `VITE_API_URL` (див. `.env.example`)
- **Версіонування:** префікс `/api/v1/...` (згідно з API-контрактом — див. Крок 9 етапу 2)

### 4.2. Шрифти
- **Georgia (serif)** — основний шрифт інтерфейсу
- **JetBrains Mono** — для заголовків секцій та фільтрів
- **Inter** — для карток та тексту

## 5. Принцип "LTS versions only"

Команда обрала стабільні версії технологій (LTS або останні стабільні релізи), а не експериментальні альфа/бета-версії. Це забезпечує:
- **Стабільність** — менше критичних багів у production.
- **Довгу підтримку** — security-патчі та виправлення ще довго після релізу.
- **Сумісність** — офіційні плагіни та бібліотеки гарантовано працюють зі стабільними версіями.
