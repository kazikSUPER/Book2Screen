# TestPlan.md

## План тестування: Система відгуків, оцінювання та каталогу книг

### 1. Вступ (Introduction)

**Мета:**
Перевірка працездатності вебсистеми для перегляду книг, пошуку, оцінювання, створення відгуків та персоналізованої взаємодії користувача з каталогом.

**Об’єкт тестування:**
До тестування входять основні функціональні модулі системи:

* **UC-01:** Реєстрація та відновлення акаунта
* **UC-02:** Керування списком «Обране»
* **UC-03:** Пошук та фільтрація каталогу
* **UC-04:** Голосування та карта розбіжностей
* **UC-05:** Створення відгуку зі спойлером

---

### 2. Scope (Що тестуємо)

#### In-Scope

У тестування входять:

* форми реєстрації та відновлення
* валідація email та пароля
* додавання / видалення книги з обраного
* пошук за назвою, автором, жанром
* робота фільтрів
* сторінка детального перегляду книги
* голосування за книгу / екранізацію
* прогрес-бари та рейтинги
* створення відгуків
* механізм приховування спойлерів
* UI переходи між екранами
* адаптивність інтерфейсу

#### Out-of-Scope

До тестування не входять:

* сторонні сервіси авторизації
* зовнішні API третіх сторін
* сторонні рекламні або аналітичні модулі

---

### 3. Стратегія тестування (Testing Strategy)

#### 3.1 Functional Testing

Перевірка відповідності функціоналу бізнес-вимогам.

Тестуються:

* реєстрація користувача
* відновлення пароля
* додавання в обране
* видалення з обраного
* пошук і фільтрація
* голосування
* створення відгуку
* відображення рейтингу
* карта розбіжностей

---

#### 3.2 UI/UX Testing

Перевірка:

* зручності навігації
* логіки переходів
* розміщення кнопок
* hover states
* loading states
* error states
* адаптивності для мобільних пристроїв

---

#### 3.3 Boundary Value Analysis

Перевірка крайніх значень вхідних даних.

Приклади:

* пароль мінімальної допустимої довжини
* максимальна довжина відгуку
* рейтинг 1 і 5
* дуже довга назва книги
* порожній результат пошуку

---

#### 3.4 Security Testing

Перевірка:

* доступу лише авторизованих користувачів до голосування
* доступу до відгуків
* захисту персональних даних
* коректної роботи прав доступу

---

#### 3.5 Integration Testing

Перевірка взаємодії:

* Frontend ↔ API
* API ↔ Database
* оновлення рейтингу після голосування
* оновлення списку «Обране»

---

### 4. Ресурси та інструменти (Tools)

* **Postman** — API testing
* **Swagger** — перевірка endpoint-ів
* **Chrome DevTools** — UI / responsiveness
* **Jest / Cypress** — frontend testing
* **Jira** — bug tracking
* **Manual Testing** — перевірка user flow

---

### 5. Entry Criteria (Критерії початку)

Тестування розпочинається, якщо:

* усі екрани системи реалізовані
* налаштовані переходи між сторінками
* створені тест-кейси для кожного Use Case
* заповнені тестові дані
* узгоджено acceptance criteria

---

### 6. Exit Criteria (Критерії завершення)

Тестування вважається завершеним, якщо:

* для кожного Use Case створено мінімум 2 тест-кейси
* виконані всі позитивні сценарії
* виконані всі негативні сценарії
* 100% critical bugs виправлено
* 100% основних User Flows проходять успішно
* відсутні blocker defects
* усі High-priority bugs закриті
* система стабільно працює на desktop і mobile

---

### 7. Risks

Можливі ризики:

* некоректна логіка переходів
* неочевидні CTA-кнопки
* дублювання функцій
* помилки валідації
* проблеми адаптивності
* конфлікти між фільтрами та пошуком

---

### 8. Schedule (План виконання)

**Етап 1:**
Тестування навігації та user flow

**Етап 2:**
Тестування функціональних Use Cases

**Етап 3:**
UI/UX review

**Етап 4:**
Regression testing

**Етап 5:**
Фінальна перевірка перед демонстрацією

---

### 9. Deliverables

* документ **TestPlan.md**
* таблиці **Test Cases для UC-01 – UC-05**
* **Bug Report / Jira Issues**
* посилання на **Figma prototype**
* **Final Test Report**
